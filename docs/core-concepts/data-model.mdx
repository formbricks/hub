## How It Works[ŌĆŗ](#how-it-works "Direct link to How It Works")

Every piece of feedbackŌĆöwhether it's an NPS score, a text comment, or a multiple-choice selectionŌĆöis stored as an individual record. This "one row per response" approach unlocks powerful capabilities:

Ō£ģ **Query with standard SQL** - No complex JSON parsing or unnesting required Ō£ģ **Connect any BI tool** - Works seamlessly with Apache Superset, Power BI, Tableau, Looker Ō£ģ **Combine data sources** - Mix surveys, reviews, support tickets in a single query Ō£ģ **Analyze trends over time** - Built-in time-series indexing for fast date-based queries Ō£ģ **AI-enriched insights** - Automatic sentiment, emotion, and topic extraction for text feedback

### Example: Survey Response[ŌĆŗ](#example-survey-response "Direct link to Example: Survey Response")

When a user completes a survey with an NPS score and a comment, Hub creates two records:

```
// Record 1: NPS score{  "source_id": "nps-2024",  "field_id": "nps_score",  "field_type": "nps",  "value_number": 9}// Record 2: Comment with AI enrichment{  "source_id": "nps-2024",  "field_id": "feedback_comment",  "field_type": "text",  "value_text": "Great service!",  "sentiment": "positive",  "emotion": "joy",  "topics": ["service_quality"]}
```

This structure makes it trivial to calculate your NPS score or find all negative sentiment across any source.

## Core Entity: FeedbackRecord[ŌĆŗ](#core-entity-feedbackrecord "Direct link to Core Entity: FeedbackRecord")

The `FeedbackRecord` entity stores individual feedback data points with optional AI enrichment for qualitative feedback.

### Field Types[ŌĆŗ](#field-types "Direct link to Field Types")

The `field_type` field uses **validated enums** to categorize responses and determine which `value_*` field to use. Hub enforces these 9 standardized types optimized for analytics:

| Type              | Description                        | Value Field     | AI Enrichment | Analytics Use                                    |
| ----------------- | ---------------------------------- | --------------- | ------------- | ------------------------------------------------ |
| **`text`**        | Open-ended qualitative feedback    | `value_text`    | Ō£ģ **Yes**     | Sentiment trends, topic analysis, word clouds    |
| **`categorical`** | Pre-defined discrete options       | `value_text`    | ŌØī No          | Frequency distribution, most popular choices     |
| **`nps`**         | Net Promoter Score (0-10)          | `value_number`  | ŌØī No          | NPS calculation, promoter/detractor segmentation |
| **`csat`**        | Customer Satisfaction (1-5 or 1-7) | `value_number`  | ŌØī No          | Average satisfaction, benchmarking               |
| **`ces`**         | Customer Effort Score (1-7)        | `value_number`  | ŌØī No          | Effort analysis, ease-of-use tracking            |
| **`rating`**      | Generic rating scale               | `value_number`  | ŌØī No          | Average rating, distribution, star ratings       |
| **`number`**      | Quantitative measurements          | `value_number`  | ŌØī No          | Sum, average, min/max, aggregations              |
| **`boolean`**     | Binary yes/no responses            | `value_boolean` | ŌØī No          | True/false counts, percentages                   |
| **`date`**        | Temporal values                    | `value_date`    | ŌØī No          | Time-series analysis, date filtering             |

<Check>
  Only `text` field types are automatically enriched with AI-powered sentiment, emotion, and topic extraction. This happens asynchronously after data is saved, optimizing costs and performance. [Learn more about AI enrichment ŌåÆ](/core-concepts/ai-enrichment)
</Check>

<Info>
  These types cover 95% of feedback management use cases while keeping the model simple. They map directly to common analytics patterns:

  * **Qualitative**: `text` ŌåÆ AI analysis for themes and sentiment
  * **Quantitative**: `nps`, `csat`, `ces`, `rating`, `number` ŌåÆ Statistical aggregations
  * **Categorical**: `categorical` ŌåÆ Frequency and distribution analysis
  * **Binary/Temporal**: `boolean`, `date` ŌåÆ Filtering and segmentation
</Info>

## Database Indexes[ŌĆŗ](#database-indexes "Direct link to Database Indexes")

Hub automatically creates indexes for optimal query performance:

* **`source_type`** - Filter by feedback source (survey, review, support)
* **`source_id`** - Query specific surveys/forms
* **`collected_at`** - Time-series queries and trending
* **`field_type`** - Filter by question type
* **`field_id`** - Group related questions across responses
* **`value_number`** - Numeric aggregations (averages, sums, counts)
* **`user_identifier`** - User-level journey analysis
* **`sentiment`** - Filter by sentiment for AI-enriched text
* **`emotion`** - Filter by emotion for qualitative analysis

<Check>
  All indexes are created automatically via database migrations. The combination of UUIDv7 primary keys and strategic indexes ensures fast queries even with millions of records.
</Check>

## UUIDv7 Primary Keys[ŌĆŗ](#uuidv7-primary-keys "Direct link to UUIDv7 Primary Keys")

Hub uses **UUIDv7** for primary keys, combining the benefits of UUIDs with time-ordered sorting:

**What you get:**

* Ō£ģ **Chronological sorting** - IDs sort by creation time automatically
* Ō£ģ **Fast database performance** - Better B-tree indexing than random UUIDs
* Ō£ģ **Globally unique** - No collision risk across distributed systems
* Ō£ģ **Horizontally scalable** - No need for central ID generation

**Example UUIDv7:**

```
01932c8a-8b9e-7000-8000-000000000001ŌööŌöĆ timestamp ŌöĆŌöś ŌööŌöĆŌöĆ random ŌöĆŌöĆŌöś
```

The timestamp prefix means newer records naturally have larger IDs, improving both query performance and ordering.

## JSONB Metadata[ŌĆŗ](#jsonb-metadata "Direct link to JSONB Metadata")

Hub uses PostgreSQL's `JSONB` type for flexible contextual data storage in the `metadata` field. This allows you to attach any custom attributes to your feedback records without schema changes.

### Common Metadata Patterns[ŌĆŗ](#common-metadata-patterns "Direct link to Common Metadata Patterns")

```
{  "metadata": {    // Device & Platform    "device": "mobile",    "os": "iOS 17.2",    "browser": "Safari 17",    "screen_size": "390x844",    // Location    "country": "US",    "region": "California",    "timezone": "America/Los_Angeles",    // Campaign & Attribution    "campaign_id": "email-001",    "referrer": "email_campaign",    "utm_source": "newsletter",    "utm_campaign": "product_launch",    // Customer Context    "customer_tier": "enterprise",    "industry": "healthcare",    "team_size": "50-200",    "plan": "pro",    // Custom Fields    "feature_flags": ["new_ui", "ai_features"],    "session_duration": 1847,    "pages_viewed": 12  }}
```

### Querying JSONB[ŌĆŗ](#querying-jsonb "Direct link to Querying JSONB")

PostgreSQL provides powerful operators for querying JSONB fields:

```
-- Extract a specific keySELECT metadata->>'country' as countryFROM feedback_recordsWHERE source_type = 'survey';-- Filter by nested valueSELECT * FROM feedback_recordsWHERE metadata->'device' @> '"mobile"';-- Check if key existsSELECT * FROM feedback_recordsWHERE metadata ? 'campaign_id';-- Filter by numeric valueSELECT * FROM feedback_recordsWHERE (metadata->>'session_duration')::int > 300;-- Array containmentSELECT * FROM feedback_recordsWHERE metadata->'feature_flags' @> '["ai_features"]';
```

<Check>
  Use snake\_case naming and consistent key names across your metadata to make queries easier:

  * Ō£ģ `"customer_tier"` (consistent)
  * ŌØī `"customerTier"`, `"tier"`, `"customer-tier"` (inconsistent)
</Check>

### Journey & Touchpoint Metadata (Recommended Convention)[ŌĆŗ](#journey--touchpoint-metadata-recommended-convention "Direct link to Journey & Touchpoint Metadata (Recommended Convention)")

For customer journey analysis, we recommend using these standardized metadata keys:

```
{  "metadata": {    "journey_stage": "onboarding",    "touchpoint": "checkout",    "channel": "mobile_app",    "interaction_type": "transaction"  }}
```

| Key                | Recommended Values                                                                     | Description                                 |
| ------------------ | -------------------------------------------------------------------------------------- | ------------------------------------------- |
| `journey_stage`    | `awareness`, `consideration`, `purchase`, `onboarding`, `usage`, `renewal`, `advocacy` | Where the customer is in their lifecycle    |
| `touchpoint`       | Free-form (e.g., `checkout`, `support_ticket`, `feature_x`)                            | Specific interaction point                  |
| `channel`          | `web`, `mobile_app`, `email`, `support`, `in_store`, `social`                          | Channel of interaction                      |
| `interaction_type` | `transaction`, `support`, `feedback_request`, `proactive`                              | Type of interaction that triggered feedback |

## Best Practices[ŌĆŗ](#best-practices "Direct link to Best Practices")

### 1. Use Validated Field Types[ŌĆŗ](#1-use-validated-field-types "Direct link to 1. Use Validated Field Types")

Always use one of the 9 standardized field types. The API will reject invalid types:

Ō£ģ **Good:**

```
{  "field_type": "text", // Valid enum  "field_type": "categorical", // Valid enum  "field_type": "nps", // Valid enum  "field_type": "ces" // Valid enum (Customer Effort Score)}
```

ŌØī **Will Fail:**

```
{  "field_type": "openText", // Old name, now invalid  "field_type": "multiple_choice", // Not a valid enum  "field_type": "custom_scale" // Not a valid enum}
```

### 2. Match Field Types to Value Columns[ŌĆŗ](#2-match-field-types-to-value-columns "Direct link to 2. Match Field Types to Value Columns")

Only populate the appropriate `value_*` field based on `field_type`:

| Field Type                               | Correct Value Column | Example                  |
| ---------------------------------------- | -------------------- | ------------------------ |
| `text`, `categorical`                    | `value_text`         | `"Great product!"`       |
| `nps`, `csat`, `ces`, `rating`, `number` | `value_number`       | `9`                      |
| `boolean`                                | `value_boolean`      | `true`                   |
| `date`                                   | `value_date`         | `"2025-01-15T10:00:00Z"` |

Ō£ģ **Good:**

```
{  "field_type": "nps",  "value_number": 9}
```

ŌØī **Avoid:**

```
{  "field_type": "nps",  "value_text": "9" // Wrong! Should be value_number}
```

### 3. Use Consistent Field IDs[ŌĆŗ](#3-use-consistent-field-ids "Direct link to 3. Use Consistent Field IDs")

Keep field IDs stable across time for longitudinal analysis:

Ō£ģ **Good:**

```
{  "field_id": "nps_score", // Consistent  "field_id": "improvement_feedback" // Consistent}
```

ŌØī **Avoid:**

```
{  "field_id": "nps_q1_2025", // Don't include time periods  "field_id": "question_1" // Too generic}
```

### 4. Leverage AI Enrichment[ŌĆŗ](#4-leverage-ai-enrichment "Direct link to 4. Leverage AI Enrichment")

For open-ended feedback, use `field_type: "text"` to get automatic sentiment, emotion, and topic extraction:

```
{  "field_type": "text",  "value_text": "The UI is great but pricing is confusing"  // AI will automatically add:  // "sentiment": "mixed"  // "sentiment_score": 0.1  // "emotion": "confused"  // "topics": ["ui_design", "pricing"]}
```

### 5. Multi-Select as Multiple Rows[ŌĆŗ](#5-multi-select-as-multiple-rows "Direct link to 5. Multi-Select as Multiple Rows")

For questions where users can select multiple options, create one row per selection:

```
// Question: "Which features do you use?" (user selected 3 options)// Create 3 rows with the same source_id, field_id, but different value_text{"field_type": "categorical", "value_text": "Dashboards"},{"field_type": "categorical", "value_text": "Reports"},{"field_type": "categorical", "value_text": "Alerts"}
```

### 6. Anonymous User Identifiers[ŌĆŗ](#6-anonymous-user-identifiers "Direct link to 6. Anonymous User Identifiers")

For privacy, use anonymous hashed IDs instead of PII:

Ō£ģ **Good:**

```
{  "user_identifier": "sha256:abc123def456"}
```

ŌØī **Avoid:**

```
{  "user_identifier": "john@example.com" // Never store PII}
```

### 7. Rich Metadata for Segmentation[ŌĆŗ](#7-rich-metadata-for-segmentation "Direct link to 7. Rich Metadata for Segmentation")

Use `metadata` for contextual attributes that enable deeper analysis:

```

```
